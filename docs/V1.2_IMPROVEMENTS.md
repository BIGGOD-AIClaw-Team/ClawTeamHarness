## 一、V1.1 遗留问题汇总

> 以下是 V1.1 计划修复但仍未解决、或 V1.1 新发现的问题。

### 1.1 阻断性问题（P0，仍未解决）

| 问题 | 现状 | 影响 |
|------|------|------|
| **LLM 调用是 Placeholder** | `LLMNode._call_llm()` 返回 echo，未接入 OpenAI/Anthropic SDK | Agent 无法真正执行 |
| **任务队列 `_execute_task` 是 TODO** | `routes/tasks.py` 的 `_execute_task()` 只是 `status = completed`，没有真实调用 Engine | 异步执行形同虚设 |
| **WebSocket 只是 Echo** | `websocket_endpoint()` 原样返回客户端消息，没有与 Agent 执行流打通 | 无法实时看到执行过程 |
| **MCP Server CRUD 未与 MCPClient 打通** | `routes/mcp.py` 能增删查配置，但 `MCPClient` 没有真实连接/调用能力 | MCP 工具无法真正使用 |
| **Skills Registry API 引用空注册表** | `SkillRegistry.list_skills()` 返回空，`skills/` 目录下没有真实 Skill 注册 | Skills 系统不可用 |
| **记忆 API 是内存 Mock** | `routes/memory.py` 用 `{}` 字典模拟，`ShortTermMemory` 等类没有真实实例化 | 记忆系统形同虚设 |
| **Agent 执行 `POST /agents/:id/execute` 是同步阻塞** | 直接 `await engine.execute()` 无超时、无中断、无法取消 | 大模型响应慢时 API 超时 |

### 1.2 重要遗留（P1）

| 问题 | 现状 |
|------|------|
| 可视化画布只有 Start/Default/End 三种节点 | 无法配置 LLM 参数、Tool 配置、Condition 表达式 |
| 意图识别器未实现 | 1.5 标记 P1，实际未交付 |
| 响应生成器未实现 | 流式输出不可用 |
| 上下文自动压缩缺失 | 长对话会撑爆内存 |
| 节点执行无超时控制 | 任意节点可无限期挂起 |
| 异常处理薄弱 | 无重试、无降级、无熔断 |
| 调试面板未实现 | 前端无执行日志/节点状态追踪 UI |
| Checkpoint 管理缺失 | 无法暂停/恢复/列举 checkpoint |
| 图执行不可中断 | 用户无法中途停止 Agent |

### 1.3 安全性问题

| 问题 | 严重性 |
|------|--------|
| `allow_origins=["*"]` CORS 全开 | P1 生产风险 |
| 无 API 鉴权 | P0 任何人都能操作 |
| 无 Rate Limiting | P0 攻击可直接压垮后端 |
| `evaluate_condition` 使用 `eval()` | P1 代码注入风险 |
| ChromaDB 无连接池 | P2 连接竞争 |

## 二、遗漏功能清单（V1.2 新增）

### 2.1 Agent 执行 Trace 系统 ⭐ 新功能

**背景**：当前 Agent 执行是一个黑箱，用户不知道节点执行顺序、每个节点的输入输出、执行时间、异常位置。

**设计目标**：完整记录 Agent 执行过程，支持事后回放和实时追踪。

#### 数据模型

```python
# Trace Record
class TraceRecord:
    trace_id: str          # 每次执行的唯一ID
    agent_id: str
    started_at: datetime
    finished_at: datetime | None
    status: Literal["running", "completed", "failed", "cancelled"]

# Node Execution Span
class NodeSpan:
    span_id: str
    trace_id: str
    node_id: str
    node_type: str         # llm / tool / condition / start / end
    status: Literal["pending", "running", "completed", "failed"]
    input_snapshot: dict   # 节点输入（脱敏后）
    output_snapshot: dict | None  # 节点输出（脱敏后）
    error: str | None
    started_at: datetime
    finished_at: datetime | None
    duration_ms: int | None
```

#### Trace 服务 API

```
GET  /api/v1/traces                    - 列出所有执行记录（分页）
GET  /api/v1/traces/:trace_id          - 获取某次执行的完整 Trace（含 NodeSpan 列表）
GET  /api/v1/traces/:trace_id/nodes/:node_id  - 获取某节点详细输入/输出
GET  /api/v1/traces/:trace_id/visualize      - 返回可视化友好的 Timeline 数据
WS   /api/v1/traces/stream/:trace_id  - 实时订阅 Trace 事件流（节点开始/结束/异常）
POST /api/v1/traces/:trace_id/cancel  - 中断正在运行的 Trace
```

#### 前端 Trace 面板

- **时间线视图**：横轴时间轴，标注每个节点 span 的起止时间
- **节点详情抽屉**：点击节点展开输入/输出 JSON，可展开/折叠
- **状态颜色**：pending=灰、running=蓝、completed=绿、failed=红
- **流式日志**：WebSocket 实时推送节点日志到前端 Console 面板
- **异常高亮**：失败节点自动放大，显示错误堆栈

#### 实现方案

1. `backend/src/traces/` 模块——TraceRecorder、TraceQuerier
2. `AgentEngine.execute()` 改造——每个节点前后记录 span，生成 trace_id
3. 内存 Ring Buffer 存储最近 1000 条 Trace（可配置），持久化到 SQLite
4. 前端 `TracePanel.tsx`——时间线 + 节点详情抽屉 + WebSocket 实时流

### 2.2 完善的日志服务系统 ⭐ 新功能

**背景**：当前后端只有 `print()` 和基础 `logging`，无结构化、无级别控制、无查询接口。

#### 日志架构

```
Logger (各模块)
    ↓ structured dict
LogProcessor (format/enrich)
    ↓
LogHandlers
    ├── ConsoleHandler  (dev)
    ├── FileHandler     (rotated daily, 保留30天)
    └── MemoryHandler  (ring buffer for API query)
```

#### 日志格式（JSON Structure）

```json
{
  "timestamp": "2026-03-29T19:56:00.123Z",
  "level": "INFO",
  "logger": "agents.engine",
  "trace_id": "abc-123",
  "task_id": "task-456",
  "agent_id": "my_agent",
  "message": "Node llm_1 started",
  "node_id": "llm_1",
  "node_type": "llm",
  "extra": {}
}
```

#### 日志级别控制

```python
# 环境变量控制
LOG_LEVEL=DEBUG           # 全局级别
LOG_LEVEL_ENGINE=INFO     # 模块级覆盖
LOG_FORMAT=json|pretty    # json机器读 / pretty人读
LOG_FILE_ENABLED=true
LOG_FILE_DIR=./logs
```

#### 日志查询 API

```
GET /api/v1/logs
    ?level=ERROR
    &logger=agents.engine
    &trace_id=abc-123
    &from=2026-03-29T00:00:00Z
    &to=2026-03-29T23:59:59Z
    &limit=100
    &offset=0

Response: {
  "logs": [...],
  "total": 1234,
  "level_counts": {"DEBUG": 10, "INFO": 500, "ERROR": 3}
}
```

#### 日志导出

```
GET /api/v1/logs/export
    ?format=jsonl|csv
    &from=...
    &to=...
    &trace_id=...
→ Content-Disposition: attachment; filename=logs_20260329.jsonl
```

#### 前端日志面板

- 独立 `/logs` 页面，实时 WebSocket 日志流
- 级别过滤器（DEBUG/INFO/WARN/ERROR）
- 关键词搜索（message 内容）
- 一键导出

## 四、Trace 功能详细设计

### 4.1 架构

```
AgentEngine.execute(input)
    ↓
trace_recorder.start_trace(agent_id, input)
    ↓
for node in graph_nodes:
    span = trace_recorder.start_span(node_id, node_type)
    try:
        result = await node.execute(state)
        span.record_output(result)
    except Exception as e:
        span.record_error(e)
    trace_recorder.end_span(span)
trace_recorder.end_trace(status)
    ↓
WebSocket broadcast: { event: "node_completed", span }
```

### 4.2 存储策略

- **内存 Ring Buffer**：保留最近 1000 条 Trace
- **SQLite 持久化**：完整历史（按日期分表）
- **Trace 数据保留期**：可配置（默认 7 天）

### 4.3 脱敏规则

- 用户输入内容仅记录长度，不记录原文
- API Key、Token 等敏感字段自动替换为 `***`
- `input_snapshot` / `output_snapshot` 支持按字段脱敏

## 五、日志服务详细设计

### 5.1 模块结构

```
backend/src/logging/
├── __init__.py
├── config.py          # 日志配置加载（环境变量）
├── formatters.py      # JSON / Pretty 格式化器
├── handlers.py         # Console / File / Memory / WebSocket Handler
├── loggers.py          # 各模块 logger 工厂
└── query.py            # 日志查询接口
```

### 5.2 日志级别对照表

| 级别 | 使用场景 |
|------|---------|
| DEBUG | 节点输入/输出详情、变量内容 |
| INFO | 节点开始/结束、执行路径 |
| WARN | 节点超时、记忆缓存满 |
| ERROR | 节点执行异常、API 调用失败 |
| FATAL | 进程崩溃、不可恢复错误 |

### 5.3 WebSocket 日志流

```
WS /api/v1/logs/stream
→ 实时推送每条日志（带 level 过滤）
→ 前端 TracePanel 内嵌展示
```

## 七、总结

V1.1 MVP **建立了系统框架**，但**核心执行链路未打通**。V1.2 的核心目标：

1. **让 Agent 真正跑起来**——LLM 集成 + 异步任务队列

3. **让用户体验合格**——节点配置面板 + 错误处理 + 暗模式

预计总工时：**13 days**（分 4 批交付，每批 2-5 days）

> 建议同步进行：后端第一批和前端画布增强可并行。
