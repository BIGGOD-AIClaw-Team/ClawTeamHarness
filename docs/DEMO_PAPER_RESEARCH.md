## 一、设计目标

将 ClawTeamHarness 从军事领域改造为**通用多智能体协作平台**，以"论文检索生成"作为示例业务场景。

### 1.1 核心转变

| 维度 | 旧设计（军事） | 新设计（通用） |
|------|---------------|---------------|
| 领域背景 | 战场情报、战术分析 | 学术研究、知识生产 |
| Agent 角色 | 情报员、分析师、指挥官 | 研究员、分析师、作家、审核员 |
| 术语语言 | 情报源、威胁评估、作战方案 | 论文数据库、相关性分析、综述报告 |
| 界面文案 | 军事风格 | 学术/专业风格 |
| 核心价值 | 快速决策、风险评估 | 信息整合、知识综述 |

### 1.2 示例业务流程

```
用户输入研究主题
       ↓
[Coordinator] 分解任务，制定计划
       ↓
[Searcher]   搜索论文数据库
       ↓
[Analyzer]    分析论文摘要，提取关键信息
       ↓
[Writer]      生成综述报告
       ↓
[Reviewer]    质量审核
       ↓
输出最终报告
```

### 2.2 Agent 间协作流程

```
User Input: "深度学习在医学影像诊断中的应用"

Coordinator
  │
  ├─→ [制定搜索计划]
  │     • 确定关键词：deep learning, medical imaging, diagnosis
  │     • 确定数据库：arXiv, PubMed
  │     • 确定数量：每库10篇
  │
  │  → Searcher
  │     • 执行搜索
  │     • 返回论文列表（标题、摘要、链接）
  │
  │  → Analyzer
  │     • 分析每篇论文摘要
  │     • 提取：方法、数据集、精度、局限性
  │     • 返回结构化分析结果
  │
  │  → Writer
  │     • 综合分析结果
  │     • 生成综述报告（引言、方法、总结）
  │
  │  → Reviewer
  │     • 检查内容完整性
  │     • 检查逻辑一致性
  │     • 检查格式规范
  │     • 返回审核意见 / 批准发布
  │
  └─→ [汇总输出]
        • 整合最终报告
        • 返回给用户
```

### 2.3 技术栈

- **前端**: React + TypeScript（通用界面）
- **后端**: Python / Node.js
- **Agent框架**: LangGraph / LangChain
- **存储**: PostgreSQL + Redis + Vector DB
- **通信协议**: WebSocket（实时状态）+ REST API

## 三、Agent 角色定义

### 3.1 Coordinator（协调 Agent）

**职能**：任务分解、进度管理、结果汇总

**Prompt 示例**：
```
你是一个专业的研究项目协调员。
你的职责：
1. 理解用户的研究主题和需求
2. 制定合理的研究计划（搜索→分析→撰写→审核）
3. 协调各 Agent 的工作节奏
4. 监控任务进度，处理异常情况
5. 汇总各 Agent 输出，形成最终报告

工作原则：
- 以用户需求为导向，不偏离主题
- 合理分配资源，避免重复劳动
- 及时反馈进度，保持透明
```

**状态定义**：
```json
{
  "status": "idle | planning | coordinating | completed | error",
  "current_phase": "search | analyze | write | review",
  "progress": 0.75,
  "agents_status": {
    "searcher": "completed",
    "analyzer": "running",
    "writer": "pending",
    "reviewer": "pending"
  }
}
```

### 3.2 Searcher（搜索 Agent）

**职能**：多数据库论文检索、去重、排序

**Prompt 示例**：
```
你是一个专业的学术搜索助手。
你的职责：
1. 根据用户主题，制定搜索策略
2. 在多个学术数据库中搜索相关论文
3. 对结果去重、排序（按相关性/时间/引用量）
4. 返回结构化的论文列表

支持的数据库：
- arXiv（arXiv.org）
- PubMed（生物医学）
- Google Scholar
- Semantic Scholar

返回格式：
{
  "papers": [
    {
      "id": "2101.12345",
      "title": "论文标题",
      "authors": ["作者1", "作者2"],
      "abstract": "摘要内容...",
      "source": "arXiv",
      "url": "链接",
      "year": 2024,
      "relevance_score": 0.95
    }
  ],
  "total_found": 50,
  "search_time": "3.2s"
}
```

**工具能力**：
- Web 搜索
- API 调用（arXiv, PubMed）
- 关键词扩展（同义词、上下位词）

### 3.3 Analyzer（分析 Agent）

**职能**：论文摘要深度分析、结构化提取、质量评估

**Prompt 示例**：
```
你是一个专业的学术分析师。
你的职责：
1. 阅读并理解论文的核心内容
2. 提取关键信息：
   - 研究问题/目标
   - 使用的方法/模型
   - 数据集和实验设置
   - 主要结论和创新点
   - 局限性和未来方向
3. 评估论文质量（方法论、实验设计、结论支撑）
4. 对比不同论文的异同点

返回格式：
{
  "paper_id": "2101.12345",
  "analysis": {
    "research_question": "研究问题",
    "method": "方法描述",
    "dataset": "数据集名称",
    "key_results": ["结果1", "结果2"],
    "limitations": ["局限1", "局限2"],
    "quality_score": 0.85
  },
  "key_insights": ["洞察1", "洞察2"]
}
```

### 3.4 Writer（撰写 Agent）

**职能**：综述报告撰写、格式规范、多版本支持

**Prompt 示例**：
```
你是一个专业的学术写作助手。
你的职责：
1. 基于分析师的结果，撰写完整的综述报告
2. 报告结构：
   - 引言（研究背景、意义）
   - 方法分类（按技术路线组织）
   - 现有工作对比（表格/图表）
   - 总结与展望
3. 确保：
   - 逻辑连贯、论述清晰
   - 引用规范、可溯源
   - 避免抄袭（改写而非复制）

输出格式支持：
- Markdown（默认）
- LaTeX
- Word（.docx）
```

### 3.5 Reviewer（审核 Agent）

**职能**：质量审核、查漏补缺、格式检查

**Prompt 示例**：
```
你是一个严格的学术审核员。
你的职责：
1. 内容审核：
   - 事实准确性
   - 逻辑一致性
   - 论证充分性
2. 完整性审核：
   - 重要论文是否覆盖
   - 关键观点是否遗漏
   - 结论是否有数据支撑
3. 格式审核：
   - 引用格式规范
   - 图表清晰度
   - 术语一致性
4. 审核结论：
   - PASS：可直接发布
   - MINOR_REVISION：需小幅修改
   - MAJOR_REVISION：需重新撰写

返回格式：
{
  "status": "PASS | MINOR_REVISION | MAJOR_REVISION",
  "issues": [
    {"type": "factual_error", "location": "第3段", "description": "...", "severity": "high"}
  ],
  "suggestions": ["建议1", "建议2"],
  "overall_quality": 0.82
}
```

## 四、论文检索生成工作流

### 4.1 完整流程图

```
用户输入研究主题
       │
       ▼
┌──────────────────┐
│   Coordinator     │
│   任务规划        │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐    ┌──────────────────┐
│    Searcher      │───→│  论文数据库1     │
│    多源搜索       │    │  (arXiv)         │
└────────┬─────────┘    └──────────────────┘
         │
         │ 论文列表（可能重复）
         ▼
┌──────────────────┐    ┌──────────────────┐
│    Searcher      │───→│  论文数据库2     │
│    去重排序       │    │  (PubMed)       │
└────────┬─────────┘    └──────────────────┘
         │
         ▼
┌──────────────────┐
│    Analyzer      │
│    摘要分析       │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│    Writer        │
│    报告撰写       │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│    Reviewer      │
│    质量审核       │
└────────┬─────────┘
         │
    ┌────┴────┐
    │  审核结果 │
    └────┬────┘
         │
    ┌────�────┐
    │ PASS?   │
    └────┬────┘
     YES │    NO（返回Writer重新撰写）
         ▼
    输出最终报告
```

### 4.2 状态机定义

```typescript
type WorkflowState =
  | { phase: 'idle'; result: null }
  | { phase: 'planning'; plan: ResearchPlan }
  | { phase: 'searching'; progress: number; papers: Paper[] }
  | { phase: 'analyzing'; progress: number; analyses: Analysis[] }
  | { phase: 'writing'; draft: Report }
  | { phase: 'reviewing'; issues: Issue[] }
  | { phase: 'completed'; report: FinalReport }
  | { phase: 'error'; error: string }
```

### 4.3 并行化策略

```
阶段内并行：
- Searcher 可同时搜索多个数据库（arXiv + PubMed + Scholar）
- Analyzer 可同时分析多篇论文

阶段间流水线：
- Searcher 找到 10 篇 → Analyzer 开始分析前 5 篇
- Analyzer 分析完 2 篇 → Writer 开始起草相关章节

反馈循环：
- Reviewer 审核不通过 → Writer 重新撰写 → Reviewer 再次审核
```

### 4.4 错误处理

| 错误场景 | 处理策略 |
|---------|---------|
| 数据库 API 超时 | 切换到备用数据源，重试 3 次 |
| 论文数量不足 | 放宽搜索条件，扩大关键词 |
| 分析失败 | 跳过该论文，标注原因 |
| 审核不通过 | 返回 Writer 修改，最多 3 轮 |
| 全部失败 | 终止流程，输出错误报告 |
